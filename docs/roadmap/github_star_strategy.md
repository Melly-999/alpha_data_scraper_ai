# GitHub Star Strategy

## Positioning Statement

Use this language consistently across the README, demo captions, and social posts:

> open-source AI trading terminal with hard no-live-trading safety posture

Avoid any language that implies guaranteed profit, financial advice, or live execution capability.

## README As Landing Page

The README should act like a landing page, not just a setup file.

It should quickly answer:

- what MellyTrade is
- why it exists
- what safety rules it follows
- what users can inspect in the UI
- how to contribute

## First 8 Seconds Rule

A visitor should understand the project’s value within 8 seconds.

Show this order:

1. title and one-line positioning
2. safety posture
3. screenshot or GIF
4. feature highlights
5. contributor hooks

## Demo GIF / Screenshot Checklist

- terminal home or dashboard shell
- read-only safety badge
- risk score widget
- backtest or walk-forward preview
- audit log feed
- AI explanation feed
- no order controls visible
- no live trading language visible

## GitHub Topics

Use topics that reinforce the project’s identity without overselling it.

Suggested examples:

- `ai`
- `trading-terminal`
- `fintech`
- `quant`
- `risk-management`
- `backtesting`
- `read-only`
- `paper-trading`
- `open-source`
- `terminal-ui`

## Repository Trust Signals

- issue templates
- PR template
- CONTRIBUTING guide
- release notes
- changelog discipline
- clear safety policy
- visible human review requirement

## Good First Issues

Focus on low-risk, high-clarity tasks such as:

- README wording improvements
- docs cleanup
- screenshot refresh
- glossary additions
- audit log labeling
- validation checklist updates

## Release Notes

Release notes should summarize:

- what changed
- why it matters
- what stayed read-only
- what safety checks were preserved

## Promotion Plan

Use low-friction, credibility-first channels:

- LinkedIn post with a short product story and screenshot
- dev.to article explaining the safety-first architecture
- Reddit post in a relevant builder or quant community if appropriate

Focus on:

- the architecture
- the validation story
- the safety posture
- the UX polish

Do not focus on profit claims.

## Safety-First Positioning

Repeat these principles consistently:

- no live trading
- no broker execution
- no order buttons
- no secrets
- no investment advice
- human review required
- paper/simulation only

## Copy Guardrails

Do not say:

- "guaranteed returns"
- "profitable strategy"
- "financial advice"
- "live execution"
- "automated trading"

Do say:

- "read-only"
- "paper-only"
- "risk-aware"
- "review-first"
- "validation-oriented"

## Contributor Growth Loop

1. publish a clean README landing page
2. add visible safety and validation docs
3. split work into small, reviewable issues
4. keep release notes frequent and useful
5. invite contributors with low-risk tasks
6. share progress through screenshots and short explainers

This approach is more credible than hype and better aligned with the project’s safety posture.
